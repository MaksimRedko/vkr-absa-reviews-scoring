# Roadmap: Путь γ. Для передачи на GPT

## Что это за документ
Это план работы на следующие 2-3 недели. Написан так, чтобы ты мог скопировать его в новый чат с GPT (или другим ассистентом), и новый ассистент понял, что мы делаем, откуда пришли, куда идём и на чём нельзя споткнуться.

Читать сверху вниз. Каждая фаза — отдельный этап, со своими критериями успеха и откатом если не получилось.

## Контекст для нового ассистента
Это дипломная работа бакалавра по теме ABSA (Aspect-Based Sentiment Analysis) для русскоязычных отзывов. Задача: по тексту отзыва определить, о каких характеристиках товара пишет пользователь, и с каким настроением.

**Ограничения, которые нельзя нарушать:**
* Только CPU. Никакого GPU, никакого fine-tuning больших моделей.
* Никаких LLM для инференса (нельзя использовать GPT-подобные модели как часть пайплайна).
* Zero-shot или близко к тому. Никаких тяжёлых обучений.
* Есть 2 месяца до защиты. Radical изменения архитектуры не принимаются.

**Что уже есть в проекте:**
* Работающий пайплайн извлечения кандидатов (KeyBERT + POS-фильтр через pymorphy3)
* Модель для NLI-инференса (`rubert-base-cased-nli-threeway`)
* Энкодер для эмбеддингов (`rubert-tiny2`)
* Байесовская агрегация (Ledoit-Wolf shrinkage)
* Размеченный датасет: 16 товаров, 1762 отзыва, 4 категории (physical_goods, consumables, hospitality, services)
* Непеременный корпус: ~1М отзывов WB + Yandex

**Что уже пробовали и отбросили:**
* Anchor-First + HDBSCAN → провалилось на Yandex из-за якорей, заточенных под WB
* Dependency-based extraction → trade-off, чистого улучшения нет
* LLM-именование кластеров → модели до 3B параметров не справляются с русским
* NLI для детекции аспектов → AUC 0.69, слабо
* Cosine от кандидата к центроиду синонимов → squashed space на `rubert-tiny2`
* Direct aspect scoring через star rating → rating шумный, плюс не на всех платформах есть звёзды

**Что выбрали — Путь γ (hybrid vocabulary):**
* Ядро словаря (~27 общих аспектов): Качество, Цена, Доставка, Упаковка и т.д. Общие для всех категорий.
* Расширения словаря под категорию (domain sub-vocabularies): для physical_goods отдельный список, для hospitality отдельный, и т.д.
* При работе используется объединение ядра и подсловаря нужной категории
Это архитектурное решение зафиксировано. Не меняем.

## Критерии успеха (зафиксированы, изменять нельзя)
* **Покрытие словаря:** на 16 размеченных товарах должно быть минимум 65% gold-аспектов попадать в словарь (ядро + подсловарь категории). Target: 75%.
* **Точность и полнота детекции:** на 16 товарах, кросс-валидация leave-one-product-out. Target: точность ≥ 0.55, полнота ≥ 0.45. Минимум: оба ≥ 0.40.
* **Ошибка сентимента:** средняя абсолютная ошибка на уровне отзыва ≤ 0.8, на уровне товара (при n≥3 отзывов) ≤ 0.6.
* **Корреляция с rating на непеременном корпусе:** на 1М неразмеченных отзывов — Spearman correlation между нашей усреднённой оценкой по аспектам и звёздами пользователя ≥ 0.6. Это внешняя валидация.
* **Защищаемость:** должны быть воспроизводимые ответы на вопросы комиссии про происхождение словаря, выбор категорий, работу на новых товарах.

## ФАЗА 1: Построить подсловари для 4 категорий
**Что делаем:**
Для каждой из 4 категорий составляем отдельный список 10-15 специфичных аспектов. Эти аспекты должны дополнять ядро, а не дублировать его.

**Принципы:**
* Берём аспекты из внешних источников: каталог WB, рубрикаторы маркетплейсов, стандартные schemas из SentiRuEval/ROMIP. НЕ смотрим в gold labels 16 размеченных товаров — это утечка.
* Если аспект уже есть в ядре (например, "Качество") — не дублируем, оставляем только в ядре.
* Каждый аспект: `id`, `canonical_name`, минимум 3-5 синонимов, `hypothesis_template`.
* Сохраняем откуда взят каждый аспект (reproducibility log).

**Конкретные исходники для категорий:**
* **physical_goods:** WB catalog → разделы "Одежда", "Товары для рукоделия", "Товары для рыбалки". Плюс SentiRuEval-2015 restaurant/laptop aspect list (адаптировать generic part).
* **consumables:** WB catalog → "Продукты питания", "Корма для животных". Плюс типичные food review aspects (вкус, запах, свежесть, состав).
* **hospitality:** booking.com review categories, Yandex Maps hotel review template. Типичные: номер, персонал, завтрак, заселение.
* **services:** SentiRuEval banking/telecom aspects, Yandex Maps service reviews. Типичные: обслуживание, ожидание, поддержка.

**Выход:**
* 4 yaml-файла в `src/vocabulary/domain/{category_id}.yaml`
* Markdown-файл reproducibility log с источниками
* Расширенный loader (умеет загружать ядро + подсловарь для категории)

**Критерий успеха фазы:**
После составления — запустить coverage analysis на gold labels 16 товаров. Должно быть ≥ 65% (минимум) или ≥ 75% (target).

**Откат если не получилось:**
Если coverage < 65% — расширяем подсловари через frequency analysis на 1М непеременного корпуса (берём топ-N частотных существительных в категории, которых ещё нет в словаре). Это формальная процедура, не in-sample tuning.

**Сколько времени:** 1-2 дня.

## ФАЗА 5: Sentiment на matched сегментах
**Что делаем:**
Для каждого (сегмент, аспект) вычисляем polarity через NLI. Две гипотезы:
* **H+:** "в этом фрагменте <аспект> оценивается положительно"
* **H-:** "в этом фрагменте <аспект> оценивается отрицательно"

`polarity = P(entailment для H+) - P(entailment для H-)`
`rating = 3 + 2 * polarity` (получаем шкалу 1-5)

Это propositional NLI task — именно тот случай, где NLI работает хорошо (в отличие от meta-гипотезы "обсуждается ли X", которая провалилась в pilot).
**Важно:** На сегменте, а не на review в целом. Короткое propositional окно — сильнее сигнал.

**Выход:**
* Модуль sentiment в `src/stages/sentiment.py` (апдейт существующего)
* Оценка MAE на 16 товарах

**Критерий успеха:** MAE на уровне отзыва ≤ 0.8.
**Сколько времени:** 1 день (почти всё уже есть, адаптируем под сегменты).

---

## ФАЗА 6: Агрегация на уровень товара
**Что делаем:**
На уровне товара собираем все (сегмент, аспект, rating) и считаем средний rating по каждому аспекту с использованием байесовской агрегации (Ledoit-Wolf shrinkage, уже есть в проекте).

**Выход:**
* Итоговая таблица: товар × аспект → rating
* MAE на уровне товара

**Критерий успеха:** product-level MAE (при n≥3) ≤ 0.6.
**Сколько времени:** 0.5 дня (используем существующий код).

---

## ФАЗА 7: Внешняя валидация на 1М корпусе
**Что делаем:**
Прогоняем всю систему на неразмеченных 1М отзывах. Для каждого товара получаем усреднённую оценку по аспектам. Сравниваем:
* Mean rating по аспектам (наша система)
* User star rating для этого товара

Считаем Spearman correlation.

**Что показывает:**
Наша система в целом согласуется с пользовательским sentiment. Это НЕ значит, что rating — ground truth. Это значит, что на больших числах наша оценка не делает систематических ошибок.

**Target:** Spearman ≥ 0.6.
**Сколько времени:** 1 день.

## ФАЗА 8: Оформление результатов для защиты
**Что делаем:**
* Все прогоны с фиксированным seed для воспроизводимости
* Ablation study: система без sub-vocabularies (только core) vs полная система. Должна показать прирост.
* Ablation: система с одним сигналом (только lexical / только cosine / только NLI) vs ensemble. Должен показать, что ensemble лучше каждого отдельно.

## 🚩 Красные флаги. Если видишь — откатываешься, не фиксишь
* Если coverage после ФАЗЫ 1 < 65% — сначала расширяем sub-vocabs через frequency analysis. Если и это не помогает — честно признаём, что универсальный словарь покрывает только X% и пишем это в ВКР.
* Если category classifier в ФАЗЕ 2 < 85% точности — используем только hardcoded mapping для 16 товаров. Для ВКР это приемлемо с disclaimer.
* Если в ФАЗЕ 4 ansemble с тремя сигналами не бьёт отдельные сигналы — сохраняем лучший из трёх как финальный метод. НЕ добавляем новые сигналы, это scope creep.
* Если в ФАЗЕ 7 correlation < 0.6 — анализируем откуда. Возможно, смещение в какой-то одной категории. Это диагностика, не fail.
* Если возникает соблазн добавить LLM / поменять архитектуру / попробовать "ещё один подход" — НЕ делаем. Фиксируем Path γ. Улучшения внутри Path γ — да. Смена архитектуры — нет.
