### Шаг 1.1: CandidateExtractor (`src/discovery/candidates.py`)

**Цель:** Из сырого текста отзыва извлечь N-грамм кандидатов, прошедших морфологический фильтр.

**Вход:** Строка — сырой текст одного отзыва.

**Выход:** Список словарей:
```python
[
    {"span": "быстрая доставка", "sentence": "Быстрая доставка, всё пришло за 2 дня", "token_indices": (0, 2)},
    {"span": "экран", "sentence": "Экран просто огонь", "token_indices": (0, 1)},
    ...
]
```

**Алгоритм:**

```
1. ПРЕДОБРАБОТКА ТЕКСТА
   1.1. Принять raw_text (строку).
   1.2. Базовая очистка:
        - Добавить пробел после знаков препинания, если его нет
          (regex: (?<=[.,!?;:])(?=[^\s]) → вставить пробел).
        - Убрать все символы кроме букв, цифр, базовой пунктуации и пробелов
          (убивает эмодзи, спецсимволы).
        - Схлопнуть множественные пробелы.
   1.3. Разбить текст на предложения.
        Эвристика: split по [.!?] с фильтрацией пустых.
        Если пунктуации нет — считать весь текст одним предложением.

2. ГЕНЕРАЦИЯ N-ГРАММ (для каждого предложения)
   2.1. Привести предложение к lowercase.
   2.2. Разбить по пробелам → список токенов.
   2.3. Сгенерировать все N-граммы для N ∈ {1, 2}:
        - Униграммы: каждый токен отдельно.
        - Биграммы: каждая пара соседних токенов.
   2.4. Отфильтровать N-граммы по длине: каждое слово ≥ min_word_length (3) символов.

3. МОРФОЛОГИЧЕСКИЙ ФИЛЬТР (pymorphy3)
   Для каждой N-граммы:
   3.1. Для каждого токена в N-грамме вызвать pymorphy3.parse(token).
   3.2. Кандидат ПРОХОДИТ фильтр, если:
        - Хотя бы один токен имеет хотя бы один разбор с POS = 'NOUN'
        - ИЛИ хотя бы один токен НЕ НАЙДЕН в словаре pymorphy3 (OOV → возможно бренд/модель)
   3.3. Кандидат НЕ ПРОХОДИТ, если все токены распознаны и ни один не имеет разбора NOUN.

4. ДЕДУПЛИКАЦИЯ
   4.1. Удалить полные дубликаты N-грамм внутри одного предложения.
   4.2. Если униграмма "доставка" и биграмма "быстрая доставка" обе прошли фильтр —
        оставить ОБЕ (KeyBERT на следующем шаге выберет лучшую).

5. ВОЗВРАТ
   Вернуть список кандидатов с привязкой к исходному предложению (нужно для шага 1.2).
```

**Стоп-слова:** Завести минимальный набор стоп-существительных, которые pymorphy3 пропустит, но аспектами они не являются:
```
{"штука", "раз", "день", "год", "вещь", "товар", "покупка", "отзыв", "человек",
 "фото", "видео", "время", "место", "вариант"}
```

**Зависимости:** `pymorphy3` (pip install pymorphy3).

**Тест (`if __name__ == "__main__"`):**
```
Входы:
  "Достоинства: Отлично очищают экран, не оставляют разводов."
  "норм пришло быстро качество ок"
  "Сдохла батарея через неделю, просто ужас"
  "Rick Owens качество огонь"

Ожидаемое поведение:
  - "экран" → проходит (NOUN)
  - "разводов" → проходит (NOUN)
  - "очищают" → НЕ проходит (VERB, нет NOUN)
  - "быстро" → НЕ проходит (ADV, нет NOUN)
  - "качество" → проходит (NOUN)
  - "батарея" → проходит (NOUN)
  - "Rick" → проходит (OOV)
  - "Owens" → проходит (OOV)
```

### Шаг 1.2: KeyBERTScorer (`src/discovery/scorer.py`)

**Цель:** Из пула кандидатов (выход шага 1.1) отобрать семантически релевантные микро-аспекты.

**Вход:**
- Список кандидатов от CandidateExtractor (со ссылкой на предложение).
- Загруженная модель rubert-tiny2.

**Выход:** Список отобранных кандидатов с score:
```python
[
    {"span": "экран", "score": 0.82, "sentence": "Экран просто огонь", "embedding": np.array(...)},
    ...
]
```

**Алгоритм:**

```
1. КОДИРОВАНИЕ ПРЕДЛОЖЕНИЙ
   1.1. Собрать уникальные предложения из всех кандидатов.
   1.2. Батчево закодировать их через rubert-tiny2 → sentence_embeddings.
        ВАЖНО: Один forward pass на предложение (не на слово!).

2. КОДИРОВАНИЕ КАНДИДАТОВ (контекстуальное)
   2.1. Для каждого кандидата:
        - Взять предложение, в котором он найден.
        - Прогнать предложение через tokenizer (без encode — взять скрытые состояния).
        - Найти позиции токенов кандидата в tokenized-последовательности.
        - Усреднить скрытые состояния (hidden states) по этим позициям.
        → Получаем КОНТЕКСТУАЛЬНЫЙ эмбеддинг спана.

   УПРОЩЁННАЯ АЛЬТЕРНАТИВА (если контекстуальные спаны слишком сложны):
   2.2. Закодировать каждый кандидат-спан как отдельную строку через model.encode().
        Это деконтекстуализированный эмбеддинг, но для rubert-tiny2 и коротких N-грамм
        разница с контекстуальным минимальна.
        → Начать с этого варианта. Контекстуальный — оптимизация на потом.

3. КОСИНУСНЫЙ СКОРИНГ
   3.1. Для каждого кандидата c в предложении s:
        score(c) = cosine_similarity(embedding(c), embedding(s))
   3.2. Отсечение: если score(c) < cosine_threshold (0.45) → отбросить.

4. MMR-ДИВЕРСИФИКАЦИЯ (внутри каждого предложения)
   4.1. Инициализировать S = {} (отобранные), R = все кандидаты предложения.
   4.2. Первый кандидат: argmax по score.
   4.3. Для каждого следующего:
        MMR(c_i) = λ · cos(c_i, s) - (1-λ) · max_{c_j ∈ S} cos(c_i, c_j)
        Выбрать c_i с максимальным MMR.
   4.4. Повторять до mmr_top_k кандидатов или пока R не пуст.
   4.5. Параметр: λ = 0.6 (из конфига).

5. ВОЗВРАТ
   Вернуть отобранные кандидаты со скорами и эмбеддингами.
```

**Важно по шагу 2:** `sentence-transformers` имеет встроенный `model.encode()`, который
делает mean pooling по всем токенам. Для кандидатов-строк это работает из коробки.
Контекстуальные спановые эмбеддинги требуют ручной работы с `model.tokenize()` +
`model.forward()` + извлечение hidden states по индексам. Это оптимизация второго приоритета.

**Зависимости:** `sentence-transformers`, `numpy`, `sklearn.metrics.pairwise.cosine_similarity`.

**Тест:**
```
Вход: Кандидаты от предыдущего шага для отзыва
  "Экран шикарный, но батарея сдохла через день. Доставка быстрая."

Ожидаемое поведение:
  - "экран" → высокий скор в контексте первого предложения
  - "батарея" → высокий скор в контексте первого предложения
  - "доставка" → высокий скор в контексте второго предложения
  - MMR должен оставить все три (они семантически далеки друг от друга)
  - Если бы были "батарея" и "аккумулятор" — MMR оставил бы один из них
```

### Шаг 1.3: AspectClusterer (`src/discovery/clusterer.py`)

**Цель:** Собрать кандидаты со ВСЕХ отзывов на один товар, кластеризовать в аспекты, дать имена.

**Вход:**
- Список ВСЕХ отобранных кандидатов (выход шага 1.2 по всем отзывам товара).

**Выход:**
```python
{
    "Качество": {
        "keywords": ["качество", "материал", "сборка", "брак"],
        "centroid_embedding": np.array(...)
    },
    "Логистика": {
        "keywords": ["доставка", "упаковка", "курьер"],
        "centroid_embedding": np.array(...)
    },
    "display_quality": {  # Микро-аспект, не попавший в якоря
        "keywords": ["экран", "матрица", "разрешение"],
        "centroid_embedding": np.array(...)
    }
}
```

**Алгоритм:**

```
1. ПОДГОТОВКА ПУЛА
   1.1. Собрать все кандидаты от всех отзывов.
   1.2. Агрегация: подсчитать частоту каждого уникального кандидата.
   1.3. c-TF-IDF фильтрация (опционально, если есть данные по нескольким товарам):
        Оставить кандидатов с высоким c-TF-IDF для данного товара.
        Если данных по другим товарам нет — пропустить этот шаг,
        использовать простой частотный фильтр: min_mentions ≥ 2.

2. КЛАСТЕРИЗАЦИЯ (HDBSCAN)
   2.1. Взять эмбеддинги всех уникальных кандидатов.
   2.2. Подать в HDBSCAN:
        - min_cluster_size = cluster_min_size (из конфига, дефолт 3)
        - metric = 'euclidean'
        - cluster_selection_method = 'eom' (excess of mass)
   2.3. Кандидаты с label = -1 (шум) → отбросить.

3. ИМЕНОВАНИЕ КЛАСТЕРОВ (Маппинг на макро-якоря)
   3.1. Для каждого кластера вычислить центроид (mean embedding).
   3.2. Нормализовать центроид.
   3.3. Посчитать cosine similarity с КАЖДЫМ якорем из словаря макро-якорей.
   3.4. Если max_similarity > anchor_similarity_threshold (0.40):
        → Имя кластера = имя якоря (для UI).
   3.5. Если ниже порога:
        → Имя кластера = слово, ближайшее к центроиду (медоид).
        → Это "новый" микро-аспект, не покрываемый словарём.

4. СЛОВАРЬ МАКРО-ЯКОРЕЙ (захардкоженный, для UI)
   Каждый якорь — список слов, которые кодируются и усредняются в один вектор.
   {
       "Цена":            ["цена", "стоимость", "деньги", "дорого", "дешево"],
       "Качество":        ["качество", "надежность", "сборка", "материал", "брак"],
       "Внешний вид":     ["дизайн", "красота", "внешность", "стиль", "цвет"],
       "Удобство":        ["удобство", "комфорт", "эргономика", "размер"],
       "Функциональность":["мощность", "скорость", "производительность", "функция"],
       "Логистика":       ["доставка", "упаковка", "курьер", "срок"],
       "Органолептика":   ["вкус", "запах", "аромат", "звук"]
   }

5. ВОЗВРАТ
   Словарь: имя аспекта → {keywords, centroid_embedding}.
```

**Зависимости:** `hdbscan`, `numpy`, `sklearn`.

**Тест:**
```
Вход: ~200 кандидатов с эмбеддингами (от 50+ отзывов на один товар).

Ожидаемое поведение:
  - HDBSCAN сформирует 4-8 кластеров.
  - Кластер со словами ["доставка", "курьер", "пришло"] → получит имя "Логистика".
  - Кластер со словами ["экран", "матрица", "дисплей"] → если нет в якорях,
    получит имя медоида (напр. "Экран").
  - Шумовые слова (label = -1) отброшены.
```

## Шаг 2: Модуль NLI Sentiment (`src/sentiment/engine.py`)

**Цель:** Определить полярность (позитив/негатив) каждого аспекта в каждом предложении.

**Вход:**
- Список пар (предложение, аспект) — берётся из привязки кандидатов к предложениям (шаг 1).

**Выход:**
```python
[
    {"review_id": "abc", "aspect": "Качество", "sentence": "...", "score": 4.7},
    {"review_id": "abc", "aspect": "Логистика", "sentence": "...", "score": 1.3},
    ...
]
```

**Алгоритм:**

```
1. ФОРМИРОВАНИЕ ПАР
   1.1. Из результатов Aspect Discovery мы знаем:
        - Какой кандидат (→ какой аспект-кластер) в каком предложении.
   1.2. Формируем список пар:
        premise = предложение
        hypothesis = "Автор доволен {имя_аспекта}" (одна гипотеза, НЕ две)

2. БАТЧЕВЫЙ ИНФЕРЕНС
   2.1. Собрать все пары в батчи по batch_size (64).
   2.2. Прогнать через NLI-модель.
   2.3. Получить для каждой пары: P_ent, P_contr, P_neut.

3. КОНВЕРТАЦИЯ В СКОР [1, 5]
   Формула:
        Score = 1 + 4 · P_ent / (P_ent + P_contr + ε)

   Интуиция:
   - P_ent ≫ P_contr → Score ≈ 5 (автор доволен)
   - P_ent ≪ P_contr → Score ≈ 1 (автор недоволен)
   - P_neut доминирует → P_ent и P_contr оба малы → Score ≈ 2.5-3.5

   Клиппинг: max(1.0, min(5.0, score))

4. ВОЗВРАТ
   Список оценок с привязкой к review_id, aspect, sentence.
```

**Ключевое отличие от v1:**
- Убран этап Presence (его роль выполняет KeyBERT из шага 1).
- Убрано скользящее окно (работаем с целым предложением).
- Одна гипотеза вместо двух (entailment vs contradiction — это уже полярность).
- Батчевый инференс вместо поштучного.

**Оценка сложности:**
- v1: ~157 500 forward passes на 500 отзывов.
- v2: ~2 000 forward passes на 500 отзывов. Ускорение ~80x.

**Зависимости:** `transformers`, `torch`.

**Тест:**
```
Пары:
  ("Экран шикарный, яркие цвета", "Качество экрана") → Score ≈ 4.5+
  ("Сдохла батарея за день", "Батарея") → Score ≈ 1.0-2.0
  ("Ну такое, средненько", "Общее впечатление") → Score ≈ 2.5-3.5
  ("Коробка мятая, но сам товар целый", "Логистика") → Score ≈ 2.0-3.0
```

## Шаг 3: Модуль AntiFraud (`src/fraud/engine.py`)

**Цель:** Рефакторинг существующего AntiFraudEngine. Логика остаётся прежней, меняется только интеграция.

**Что менять:**
- Перенести из `app/core/anti_fraud.py` в `src/fraud/engine.py`.
- Изменить путь загрузки модели: использовать `cache_folder` из конфига.
- Добавить логарифмическую субаддитивную агрегацию для кластеров ботов (из Architecture_v1):

**Вход:** Список текстов отзывов.

## Шаг 4: Математическое ядро (`src/math/engine.py`)

**Цель:** Рефакторинг RatingMathEngine с тремя изменениями.

### Изменение 4.1: Нейтральный априор

**Было:**
```
bayesian_score = (v/(v+m)) * weighted_mean + (m/(v+m)) * global_avg_rating
```

**Стало:**
```
C = min(median_mentions, C_max)     # C_max = 3 из конфига
prior = 3.0                         # Нейтральный MaxEnt априор
bayesian_score = (N_k/(N_k+C)) * weighted_mean + (C/(N_k+C)) * prior
```

Где N_k — число упоминаний аспекта k, взвешенных по trust_weight.

### Изменение 4.2: Ledoit-Wolf ковариация

**Было:** Дисперсия каждого аспекта считается независимо.

**Стало:**
1. Построить матрицу наблюдений X: строки = отзывы (только те, где ≥ 2 аспектов),
   столбцы = аспекты, значения = оценки.
2. Применить `sklearn.covariance.LedoitWolf().fit(X)` → получить shrinkage-матрицу Σ.
3. Использовать Σ для расчёта portfolio variance при персонализации.

**Если отзывов с ≥ 2 аспектами < 5:** использовать диагональную матрицу (fallback).

### Изменение 4.3: Дисперсионный штраф → UI-алерт

**Было:** `final_score = bayesian_score - (variance_penalty * std_dev)`

**Стало:** `variance_penalty = 0.0` (из конфига). Дисперсия сохраняется в выходных данных
как поле `controversy` для отображения UI-алерта ("Мнения расходятся"), но не влияет на скор.

**Тест:**
```
Вход: Как в текущем math_engine.py __main__ (бот + старый отзыв + свежий отзыв).
Ожидание:
  - Бот (fraud_weight=0.01) → отсечён.
  - "Качество" при малом числе упоминаний → подтянуто к 3.0, НЕ к 4.0.
  - Поле "controversy" присутствует, но не влияет на "score".
```

## Шаг 5: Оркестратор (`src/pipeline.py`)

**Цель:** Связать модули 1-4 в единый пайплайн. Заменяет `app/services/analyzer.py`.

**Алгоритм верхнего уровня:**

```
def analyze_product(nm_id, limit=100):

    # 1. Загрузка данных
    reviews = loader.load_reviews(nm_id, limit)
    texts = [r.full_text for r in reviews if r.full_text]  # Только full_text

    # 2. AntiFraud → веса доверия
    trust_weights = fraud_engine.calculate_trust_weights(texts)

    # 3. Aspect Discovery (для ВСЕХ отзывов разом)
    all_candidates = []
    review_candidates_map = {}  # review_id → [candidates]

    for i, text in enumerate(texts):
        candidates = candidate_extractor.extract(text)
        all_candidates.extend(candidates)
        review_candidates_map[reviews[i].id] = candidates

    # 4. KeyBERT-скоринг + MMR
    scored_candidates = keybert_scorer.score_and_select(all_candidates)

    # 5. Кластеризация → Аспекты
    aspects = aspect_clusterer.cluster(scored_candidates)
    aspect_names = list(aspects.keys())

    # 6. NLI Sentiment → Оценки аспектов per review
    #    Формируем пары (sentence, aspect) из привязки кандидатов
    sentiment_pairs = build_sentiment_pairs(scored_candidates, aspects)
    sentiment_scores = sentiment_engine.batch_analyze(sentiment_pairs)

    # 7. Математическая агрегация
    #    Собираем: review_id → {aspect: score, fraud_weight, date}
    aggregation_input = build_aggregation_input(
        reviews, sentiment_scores, trust_weights
    )
    final_results = math_engine.aggregate(aggregation_input)

    return final_results
```

**Ключевой момент:** Функция `build_sentiment_pairs` — связующее звено.
Она берёт кандидатов (которые привязаны к предложениям И к кластерам-аспектам)
и формирует список (sentence, aspect_name) для NLI.

**Тест:** End-to-end на 2-3 товарах из разных доменов. Сравнить с результатами v1.
