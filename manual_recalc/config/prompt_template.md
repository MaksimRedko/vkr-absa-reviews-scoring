Ты помогаешь с предварительным черновиком ручной разметки ABSA.

Твоя задача: по каждому отзыву сопоставить аспекты, найденные системой, с gold-аспектами ручной разметки.

Это НЕ финальная разметка. Это только черновое предложение для последующей проверки человеком.

Верни только валидный JSON без пояснений вне JSON.

Формат ответа:

```json
{
  "batch_id": "...",
  "items": [
    {
      "review_id": "...",
      "system_aspect_decisions": [
        {
          "system_aspect": "...",
          "manual_decision": "TP | FP | UNCLEAR | DUPLICATE",
          "mapped_gold_aspect": "... | null",
          "manual_sentiment_decision": "OK | TOO_LOW | TOO_HIGH | WRONG_POLARITY | NOT_EVALUATED",
          "comment": "..."
        }
      ],
      "gold_aspect_decisions": [
        {
          "gold_aspect": "...",
          "status": "FOUND | FN | UNCLEAR",
          "matched_system_aspect": "... | null",
          "comment": "..."
        }
      ]
    }
  ]
}
```

Правила сопоставления аспектов:

1. Смотри на смысл системного аспекта в конкретном отзыве, а не только на его название.
2. Если системный аспект по смыслу соответствует одному из gold-аспектов, ставь:
   `manual_decision = "TP"`
   `mapped_gold_aspect = название соответствующего gold-аспекта`
3. Если системный аспект явно не обсуждается в отзыве или не соответствует ни одному gold-аспекту, ставь:
   `manual_decision = "FP"`
   `mapped_gold_aspect = null`
4. Если системный аспект реально есть в отзыве, но отсутствует в gold-разметке, ставь:
   `manual_decision = "UNCLEAR"`
   `mapped_gold_aspect = null`
   `comment = "Аспект есть в тексте, но отсутствует в gold"`
5. Если несколько системных аспектов покрывают один и тот же gold-аспект, только один из них помечай как `TP`, остальные помечай как `DUPLICATE`.
6. Не придумывай новые gold-аспекты. `mapped_gold_aspect` может быть только из gold-списка данного отзыва или `null`.
7. Не маппь всё подряд в общий аспект "Качество". Маппь в "Качество" только если в тексте реально обсуждается качество товара, материал, сборка, исправность, надёжность или общее исполнение.
8. Не маппь доставку, упаковку, цену, внешний вид или размер в "Качество", если они явно являются отдельными смыслами.

Правила для `gold_aspect_decisions`:

1. Для каждого gold-аспекта проверь, был ли он покрыт хотя бы одним системным аспектом.
2. Если покрыт, ставь:
   `status = "FOUND"`
   `matched_system_aspect = соответствующий системный аспект`
3. Если не покрыт, ставь:
   `status = "FN"`
   `matched_system_aspect = null`
4. Если покрытие спорное, ставь:
   `status = "UNCLEAR"`

Правила оценки тональности:

1. Оценивай тональность только если `manual_decision = "TP"`.
2. Если системный аспект `FP`, `UNCLEAR` или `DUPLICATE`, ставь:
   `manual_sentiment_decision = "NOT_EVALUATED"`
3. Если оценка системы примерно соответствует gold-оценке, ставь `OK`.
4. Если система занизила оценку, но не перевернула смысл, ставь `TOO_LOW`.
5. Если система завысила оценку, но не перевернула смысл, ставь `TOO_HIGH`.
6. Если система перепутала позитив и негатив, ставь `WRONG_POLARITY`.
7. Если сомневаешься, ставь `NOT_EVALUATED`, а не выдумывай уверенное решение.

Дополнительные правила:

1. Используй только данные из `Batch data`.
2. Не добавляй отзывы, которых нет в `Batch data`.
3. Не удаляй отзывы из `Batch data`.
4. Для каждого `review_id` из `Batch data` должен быть один `item` в ответе.
5. Для каждого системного аспекта из `Batch data` должно быть одно решение в `system_aspect_decisions`.
6. Для каждого gold-аспекта из `Batch data` должно быть одно решение в `gold_aspect_decisions`.
7. `comment` должен быть коротким: одна фраза, почему принято решение.
8. Если решение очевидно, `comment` можно оставить пустым.
9. Ответ должен быть строго валидным JSON.
