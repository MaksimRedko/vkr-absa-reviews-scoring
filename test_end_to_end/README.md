# `test_end_to_end`

Это отдельный честный demo на 20 отзывах.

Что здесь показывается:
- один и тот же набор аспектных сущностей создаётся один раз
- сохраняются отдельные сущности:
  - `reviews.jsonl`
  - `aspects.jsonl`
  - `fragments.jsonl`
  - `aspect_assignments.jsonl`
- потом режимы `A/B/C` восстанавливаются только из этих сущностей
- повторный поиск аспекта по тексту не делается

Минимальная идея:
- отзыв хранится отдельно
- аспект в отзыве хранится как назначение на фрагмент
- фрагмент хранит `review_id + start_offset + end_offset`
- потом:
  - `A` берёт весь отзыв
  - `B` берёт предложение
  - `C` берёт окно

Запуск:

```powershell
.venv\Scripts\python.exe test_end_to_end\demo_pipeline.py
```

После запуска артефакты лежат в:

```text
test_end_to_end/generated/
```

Смысл этого demo:
- показать схему, при которой `A/B/C` стартуют из одной и той же базы
- показать, какие именно сущности надо сохранять для честных A/B прогонов
