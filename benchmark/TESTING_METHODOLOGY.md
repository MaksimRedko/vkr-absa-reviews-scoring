# Методика тестирования, кластеризация, тональность и метрики

Документ описывает, как в репозитории устроены **кластеризация аспектов**, **NLI-тональность**, **поток данных до метрик** и какие **числа** зафиксированы в последнем прогоне бенчмарка (`benchmark/eval_datasets/results/benchmark_results.json`). Базовая конфигурация задаётся в `configs/configs.py` (OmegaConf).

---

## 1. Какая кластеризация используется при текущем конфиге

### 1.1. Рабочий путь пайплайна: не «дивизионная», а якорная + остаток

Класс `ABSAPipeline` (`src/pipeline.py`) по умолчанию подставляет **`AspectClusterer`**:

```python
# src/pipeline.py — по умолчанию
self.clusterer = clustering_stage or AspectClusterer(model=self._encoder)
```

Класс **`DivisiveClusterer`** (`src/stages/clustering.py`) реализует рекурсивное деление в UMAP-пространстве (вариант «дивизионного» дерева), но **нигде в оркестраторе не подключается** — это отдельная реализация для экспериментов. В типичном и бенчмарковом прогоне участвует только **`AspectClusterer`**.

Схема **`AspectClusterer`** (кратко, как в шапке модуля):

1. **Pass 1 — якорное назначение:** эмбеддинги унифицированных спанов сравниваются с **макро-якорями** `MACRO_ANCHORS` (словарь якорь → список фраз). Для каждого спана берётся лучший якорь по косинусу, но спан попадает в якорный кластер только если  
   `best_sim ≥ anchor_assign_threshold` **и** `best_sim − second_best ≥ anchor_assign_margin`.  
   Пороги по умолчанию в коде: `0.55` и `0.03` (через `getattr(config.discovery, ...)` — в `configs.py` отдельных ключей нет).

2. **Фильтр мусора:** спаны с избыточной близостью к **anti-якорям** `ANTI_ANCHORS` относительно макро-якорей отбрасываются в остаток (`anti_anchor_threshold` в конфиге).

3. **Pass 2 — остаток:** если остаточных спанов **≥ 5**, для них строится **UMAP** (параметры `umap_*` из `config.discovery`), затем **HDBSCAN** (`hdbscan_min_samples` из конфига; `min_cluster_size` адаптивный). Получившиеся кластеры **именуются** (см. §2), мержатся в UMAP-пространстве при евклидовом расстоянии центроидов **≤ `cluster_merge_threshold`** (0.95).

4. **Pass 3 — `min_mentions`:** после слияний остаются аспекты с суммарным числом упоминаний спанов **≥ `min_mentions`** (в вызове из пайплайна по умолчанию **2**).

Итого при данном конфиге: **якорная** привязка спанов к фиксированным тематическим группам + **кластеризация остатка** (UMAP + HDBSCAN), а не чисто дивизионная схема.

## 2. Наименование кластеров и поле `nli_label`

### 2.1. Имена после Pass 1 (уверенные якорные кластеры)

Ключ словаря аспектов — **строковое имя якоря** из `MACRO_ANCHORS` (например, «Цена», «Качество»). Это не LLM-именование: имя задано словарём.

### 2.2. Имена residual-кластеров (`_name_cluster`)

Для каждого HDBSCAN-кластера считается центроид эмбеддингов спанов. Далее:

- Считается косинусная близость центроида ко **всем макро-якорям**.
- Если **margin** между лучшим и вторым якорем **≥ `anchor_similarity_threshold`** (в `configs.py` это ключ с комментарием про residual; значение **0.05**), кластер **называется именем лучшего якоря** (флаг «medoid-имя» = false).
- Иначе имя кластера — **текст спана-медоида** (спан из кластера с максимальной косинусной близостью к центроиду; флаг medoid = true). Так появляются «сырые» имена вроде фрагментов n-грамм.

### 2.3. `nli_label` для NLI-гипотез

После фильтрации вызывается `_apply_nli_labels`:

- Если кластер **не** помечен как medoid-именованный, **`nli_label = имя кластера`** (совпадает с якорем).
- Если кластер **medoid**, **`nli_label` переназначается** на якорь с **максимальным** `cos(centroid, anchor_embedding)` среди `MACRO_ANCHORS`.

Так обеспечивается **remap**: даже «странное» имя кластера для UI/ключей сопоставляется с **каноническим якорём** для шаблонов вида «`{aspect} — это хорошо`».

**Важно:** классы `MedoidNamer` / `LLMNamer` / `GigaChatNamer` в `src/stages/naming.py` в стандартном `ABSAPipeline` **не вызываются**; переименование через LLM — отдельный контур, не задействованный в текущем baseline-пайплайне.

## 3. Тональность: одна или две гипотезы

Используется **`SentimentEngine` v3** (`src/stages/sentiment.py`) — **всегда две встречные гипотезы** на каждую пару (premise, аспект):

- `H_pos = hypothesis_template_pos.format(aspect=...)`
- `H_neg = hypothesis_template_neg.format(aspect=...)`

В `configs/configs.py` зафиксирован **вариант B**:

- позитив: `{aspect} — это хорошо`
- негатив: `{aspect} — это плохо`

Под `{aspect}` подставляется **`nli_label`** (для medoid-кластеров — уже переназначенный якорь, см. §2.3).

Модель NLI прогоняется **дважды** на батч (объединённые premise + pos-hyp и premise + neg-hyp), из логитов берётся вероятность **entailment**, затем скор в шкале 1…5 строится из отношения `P_ent_pos` и `P_ent_neg` (см. формулу и комментарии в `sentiment.py`). Применяется **temperature** на логитах (`config.sentiment.temperature`, сейчас **0.7**).

### 3.1. Уровень гранулярности: pairing strategy

В конфиге используется **`pairing_strategy`**. Текущий основной режим — `review_provenance`: пары для NLI строятся только для тех аспектов, в которые реально попали кандидаты данного отзыва. Legacy-флаг `review_level` оставлен только как fallback для старых конфигов; новые гипотезы должны переключаться через `pairing_strategy`.

### 3.2. Пост-фильтр релевантности

После NLI пары отбрасываются, если  
`p_ent_pos + p_ent_neg < relevance_threshold` (**0.6** в конфиге).

## 4. Поток данных: от датасета до метрик

Ниже — путь, который использует бенчмарк-скрипт `benchmark/run_benchmark.py` и общий код `eval_pipeline.py`.

1. **Вход:** CSV с разметкой (например `benchmark/eval_datasets/combined_benchmark.csv`; в сохранённом результате указан абсолютный путь к машине прогона).
2. **`load_markup`:** чтение CSV, парсинг колонки `true_labels` в словарь аспект → оценка.
3. **`run_pipeline_for_ids`:** для каждого `nm_id` загрузка отзывов из CSV, прогон `ABSAPipeline` (AntiFraud → кандидаты → KeyBERT+MMR → **AspectClusterer** → NLI → агрегация). На выходе — предсказанные аспекты, `per_review` (оценки по аспектам на отзыв), `aspect_keywords`.
4. **Маппинг pred → true** (для сравнения с разметкой):
   - режим **`auto`** (как в сохранённом `benchmark_results.json`): жадное сопоставление по косинусу между эмбеддингом предсказанного аспекта (среднее по keywords) и эмбеддингами эталонных описаний **`ASPECT_DESCRIPTIONS`** из `eval_pipeline.py`; порог **0.3** (`--auto-threshold`).
   - режим **`manual`**: таблица `MANUAL_MAPPING` в `eval_pipeline.py`.
5. **`evaluate_with_mapping`:** precision/recall по множествам аспектов, mention recall (в т.ч. per-review), sentence-level MAE (raw и LOO-калибровка по товарам).
6. **`evaluate_product_ratings`:** MAE по **агрегированным** оценкам аспекта на уровне товара (matched sample), в т.ч. вариант с фильтром `n_true ≥ 3`.

**Сид:** в `run_benchmark.py` по умолчанию `set_global_seed(42)`.

## 5. Определения метрик (кратко)

- **Precision (per product):** доля предсказанных аспектов, для которых в маппинге нашлась ненулевая истинная метка (`mapped_pred / |pred|`). В сохранённом прогоне **macro_precision = 1.0** — все предсказанные имена были сопоставлены с каким-то true-аспектом при `auto` и τ=0.3 (при другом пороге/датасете значение может упасть).
- **Recall (per product):** доля истинных аспектов товара, которые попали под хотя бы одно предсказание после маппинга (`|mapped true| / |true|`).
- **Mention recall (review):** доля пар (отзыв, истинный аспект), для которых в этом отзыве есть предсказанный скор хотя бы по одному предикту, сопоставленному с данным true-аспектом. **Global** усредняет по всем таким упоминаниям в датасете.
- **Sentence-level MAE (raw):** среднее |pred − true| по всем сопоставлённым парам скоров на уровне отзывов; **calibrated** — та же ошибка после линейной калибровки, обученной LOO по другим товарам (`_fit_calibration` в `eval_pipeline.py`).
- **Product-level MAE:** для каждого true-аспекта сравниваются средние оценки (true vs pred) по подмножеству отзывов с предсказанием; затем усреднение по аспектам/товарам. **`*_filtered`** — только аспекты с `n_true ≥ 3` в разметке.

Подробные комментарии в коде: `evaluate_with_mapping`, `evaluate_product_ratings` в `eval_pipeline.py`.
