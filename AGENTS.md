\# Дипломный проект ABSA



\## Контекст

Это дипломный проект по аспектно-ориентированному анализу отзывов (ABSA) для русскоязычных отзывов.

Ограничения:

\- Только CPU

\- Без GPU

\- Без fine-tuning больших моделей

\- Без LLM в production inference

\- Нужны воспроизводимые эксперименты и честные метрики



\## Текущий рабочий курс

Главный документ проекта: `NewRoadMap.txt`.



Рабочая тактика:

1\. Сначала vocabulary

2\. Потом baseline matching на текущем extractor

3\. Потом A/B: candidates vs segmentation

4\. Потом A/B: single-signal vs ensemble matching

5\. Category classifier откладывается до появления рабочего matching

6\. Sentiment stage не ломать без отдельного обоснования



\## Жёсткие правила работы

\- Нельзя менять несколько архитектурных переменных сразу

\- Каждое изменение должно быть изолированным экспериментом

\- Перед изменением кода агент обязан:

&#x20; 1. написать краткий план

&#x20; 2. указать, какая гипотеза проверяется

&#x20; 3. перечислить файлы, которые собирается менять

&#x20; 4. указать, какая метрика должна измениться

\- Если нет явной метрики и гипотезы, код не менять



\## Правила для экспериментов

\- Каждый эксперимент должен иметь:

&#x20; - имя

&#x20; - краткую цель

&#x20; - baseline

&#x20; - изменяемую переменную

&#x20; - ожидаемый эффект

\- Не трогать одновременно vocabulary, segmentation, matching и sentiment

\- Если baseline уже известен как слабый, использовать его как reference point, а не как финальное решение



\## Приоритеты

1\. Coverage vocabulary

2\. Review-level precision/recall/F1 по aspect detection

3\. MAE sentiment на review level

4\. Product-level aggregation



\## Поведение агента

\- Сначала анализировать, потом предлагать изменения

\- Для крупных задач сначала создавать/обновлять план в `.opencode/plans/current.md`

\- После каждого шага писать:

&#x20; - что сделано

&#x20; - что изменилось

&#x20; - что проверять дальше

\- Не делать скрытых рефакторингов "по пути"

\- Не переписывать рабочие части пайплайна без необходимости
\- Не использовать живой поиск по маркетплейсам и коммерческим сайтам для построения vocabulary.
\- Предпочитать воспроизводимые стабильные источники.
\- Если внешний поиск всё же нужен, сначала перечислить источники и получить подтверждение.
\- Если category слишком широкая, сначала предложить candidate aspect families, а не сразу писать YAML.



\## Что сейчас не делать

\- Не строить category classifier

\- Не переписывать sentiment engine

\- Не встраивать LLM naming

\- Не менять unit of analysis без явного A/B сравнения
