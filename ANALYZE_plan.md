## 0. Контекст и цели анализа

Проект представляет собой дипломную работу (ВКР) по теме автоматического анализа аспектов и тональности отзывов с маркетплейса Wildberries. Пайплайн включает 7 последовательных модулей: загрузка данных → фрод-детекция → морфологическая экстракция → семантический ранжировщик → кластеризация аспектов → NLI-сентимент → байесовская агрегация.

**Цели анализа:**
1. Оценить корректность архитектурных решений каждого модуля
2. Верифицировать метрики оценки (Precision / Recall / MAE)
3. Найти узкие места и потери через воронку пайплайна
4. Убедиться в воспроизводимости экспериментов
5. Выработать рекомендации по улучшению для финальной версии ВКР

## 7. Анализ NLI-сентимента (`src/sentiment/engine.py`)

### 7.1 Dual-hypothesis подход
- [ ] Проверить шаблоны гипотез:
  - H+: `"Автор доволен {aspect}"`
  - H-: `"Автор недоволен {aspect}"`
- [ ] Убедиться, что используется `entailment`-вероятность, а не `neutral`
- [ ] Проверить формулу оценки: `score = 1 + 4 * P(ent|H+) / (P(ent|H+) + P(ent|H-) + ε)`
  - Диапазон: [1.0, 5.0] ✓ (при ε → 0)
  - При P(H+) = P(H-): score = 3.0 ✓ (нейтральный)
  - При P(H+) >> P(H-): score → 5.0 ✓

### 7.2 Батчевая обработка
- [ ] Проверить размер батча (`batch_size` в конфиге) — оптимален ли для GPU/CPU
- [ ] Убедиться, что модель переносится на GPU при наличии (`device` detection)
- [ ] Проверить обработку длинных текстов: токенизация с `max_length` и `truncation=True`

### 7.3 Контекст для NLI
- [ ] Проверить: какой текст подаётся как premise — `full_text`, `pros`, или конкатенация?
- [ ] Оценить влияние длины premise на качество NLI

## 8. Анализ математической агрегации (`src/math/engine.py`)

### 8.1 Байесовская агрегация
- [ ] Проверить формулу:
  ```
  score = (N / (N + C)) * weighted_mean + (C / (N + C)) * prior
  ```
  - Prior = 3.0 (нейтральный, MaxEnt для 5-балльной шкалы) ✓
  - C = min(median_mentions, C_max) — проверить значение C_max в конфиге
- [ ] Убедиться, что `weighted_mean` использует trust_weights из AntiFraudEngine
- [ ] Проверить поведение при N=1 (единственный отзыв) → score должен быть близок к prior

### 8.2 Time Decay
- [ ] Проверить формулу экспоненциального затухания: `w_t = exp(-λ * days_since_review)`
- [ ] Убедиться, что λ (decay rate) вынесен в конфиг
- [ ] Проверить обработку отзывов без `created_date` (NaT/None)

### 8.3 Controversy Score
- [ ] Убедиться, что controversy = std_dev взвешенных оценок (не variance)
- [ ] Проверить порог в UI для отображения предупреждения
- [ ] Проверить, что `variance_penalty = 0.0` и итоговый score не занижается

### 8.4 Ledoit-Wolf ковариационная матрица
- [ ] Убедиться, что матрица рассчитывается корректно при малом числе наблюдений (N < p)
- [ ] Проверить, используется ли ковариация в итоговой оценке или только как диагностика
- [ ] Проверить обработку вырожденных случаев: N < 2 аспектов
